Three.js 射箭对抗游戏 MVP 需求文档

1. 项目概述

本项目旨在使用 Three.js 开发一个网页端 3D 射箭对抗游戏的 MVP（最小可行产品）版本。核心目标不是实现完整“吃鸡”玩法，而是验证以下核心体验是否成立：
	•	小型 3D 场景中的移动与掩体博弈
	•	弓箭作为唯一武器的射击手感
	•	玩家利用墙体、石头、树木进行躲避与输出
	•	基础战斗循环是否具有可玩性

本阶段聚焦于：
	•	静态小地图
	•	单机玩法
	•	简单掩体
	•	弓箭射击
	•	基础角色动作（蹲下、探头）
	•	血量系统
	•	基础敌人 AI

不包含多人联机、缩圈、搜物资、复杂背包、角色成长、复杂动画系统等扩展内容。

⸻

2. 产品目标

2.1 核心目标

验证“第三人称小地图掩体射箭对抗”这一玩法是否成立。

2.2 体验目标

玩家应当能够：
	•	在小地图中自由移动并寻找掩体
	•	借助墙、石头、树木隐藏自身
	•	通过蹲下与左右探头降低暴露面积
	•	使用弓箭进行瞄准、蓄力和发射
	•	与一个或多个基础 AI 敌人进行对抗
	•	通过击败敌人或生存获胜

2.3 MVP 成功标准

满足以下条件即可视为 MVP 达标：
	•	玩家可稳定移动、转向、观察场景
	•	玩家可完成射箭并命中目标
	•	掩体遮挡能影响是否被命中
	•	蹲下与探头可用于战斗博弈
	•	敌人具备基础巡逻 / 发现 / 攻击行为
	•	存在明确胜负反馈

⸻

3. 目标平台与技术范围

3.1 目标平台
	•	平台：PC 浏览器优先
	•	操作方式：键盘 + 鼠标
	•	首版暂不考虑移动端触控适配

3.2 技术栈建议
	•	渲染：Three.js
	•	开发语言：TypeScript 或 JavaScript
	•	构建工具：Vite
	•	模型格式：glTF / GLB
	•	碰撞 / 简单物理：
	•	MVP 可先使用自定义基础碰撞
	•	如需更稳定，可选 Rapier 或 cannon-es
	•	UI：HTML/CSS 覆盖层
	•	动画：Three.js AnimationMixer

3.3 非目标范围

以下内容不属于 MVP：
	•	多人联机
	•	大地图
	•	缩圈机制
	•	物资拾取与背包
	•	多武器系统
	•	技能系统
	•	高级物理破坏
	•	复杂角色定制
	•	正式商业级美术表现

⸻

4. 游戏定位

4.1 类型定位
	•	第三人称 3D 动作 / 射击原型
	•	战术掩体对抗
	•	武器限定为弓箭

4.2 核心卖点
	•	不做枪战，转而强调弓箭的蓄力、下坠与预判
	•	掩体战强调蹲下、探头、时机判断
	•	小地图降低开发复杂度，突出战斗核心

4.3 摄像机视角

MVP 采用：
	•	第三人称越肩视角

原因：
	•	便于观察掩体与周边环境
	•	更适合实现蹲下与探头玩法
	•	更容易体现“战术对抗”而非纯靶场射击

⸻

5. 核心玩法循环

玩家进入小型地图后，通过移动寻找掩体，与敌方单位周旋，利用蹲下、探头和射箭机制击败敌人，同时避免自身被击中。核心循环如下：
	1.	寻找掩体或移动到有利位置
	2.	观察敌人位置
	3.	进入瞄准状态
	4.	选择直接射击、蹲射或探头射击
	5.	发射弓箭
	6.	命中则削减敌方血量，未命中则重新调整位置
	7.	敌人反击，玩家继续利用掩体规避
	8.	击败全部敌人或被击败，回合结束

⸻

6. 功能需求

6.1 玩家控制

6.1.1 移动
	•	使用 WASD 控制角色移动
	•	角色朝向与摄像机朝向协同
	•	支持站立状态下的前后左右移动

6.1.2 视角控制
	•	鼠标控制镜头旋转
	•	镜头需跟随角色
	•	镜头需避免严重穿墙
	•	镜头在掩体附近应尽量保持可见性

6.1.3 蹲下
	•	按键切换蹲下状态（建议 Ctrl 或 C）
	•	蹲下后角色高度降低
	•	蹲下后移动速度下降
	•	蹲下状态应影响受击面积 / 可见面积

6.1.4 左右探头
	•	Q：左探头
	•	E：右探头
	•	探头时角色主体基本不移动，摄像机和射击起点横向偏移
	•	探头应可与掩体交互，形成“只露出部分身体”的效果
	•	探头状态下允许射击

6.1.5 状态约束
	•	探头时不可进行大幅移动
	•	蹲下与探头允许组合，但优先先做站立探头，再扩展蹲下探头
	•	瞄准 / 蓄力 / 发射过程中的状态切换要有限制，避免动作冲突

⸻

6.2 弓箭战斗系统

6.2.1 武器限定
	•	玩家唯一武器为弓箭
	•	不存在枪械、近战、投掷物等其他武器

6.2.2 瞄准
	•	右键进入瞄准模式
	•	瞄准模式下：
	•	摄像机适度拉近
	•	角色移动速度降低
	•	准星显示

6.2.3 蓄力
	•	左键按住开始蓄力
	•	蓄力时间影响箭速 / 射程 / 下坠表现
	•	需设置最小与最大蓄力值

6.2.4 发射
	•	松开左键发射箭
	•	箭为可见飞行物，不是瞬时命中
	•	箭飞行受重力影响，形成抛物线
	•	箭头方向应尽量贴合速度方向

6.2.5 命中判定
	•	箭与敌人、掩体、地面发生碰撞时触发结果
	•	命中敌人：造成伤害
	•	命中掩体 / 地面：停止飞行并保留短暂视觉效果，或直接销毁

6.2.6 伤害规则
	•	命中敌人扣血
	•	MVP 阶段可统一为固定伤害
	•	头部 / 身体分区伤害可暂不做，或列为增强项

⸻

6.3 血量与胜负系统

6.3.1 玩家血量
	•	玩家拥有基础生命值
	•	受到箭伤后扣血
	•	血量为 0 时判定失败

6.3.2 敌人血量
	•	敌人拥有基础生命值
	•	血量为 0 时死亡或失去行动能力

6.3.3 胜利条件
	•	击败场上全部敌人

6.3.4 失败条件
	•	玩家生命值归零

6.3.5 回合结束反馈
	•	显示胜利 / 失败界面
	•	支持重新开始

⸻

6.4 地图与掩体系统

6.4.1 地图规模
	•	小型静态地图
	•	面积控制在便于短时对抗的范围内
	•	不考虑开放世界或大范围探索

6.4.2 场景构成
地图中至少包含以下静态掩体类型：
	•	墙体
	•	石头
	•	树木

6.4.3 掩体要求
	•	掩体需要有实体碰撞
	•	掩体能够遮挡箭矢
	•	掩体高度应覆盖站立与蹲下两种博弈情况
	•	至少存在几个可以触发“左右探头”的典型掩体点位

6.4.4 地图表现要求
	•	不要求高精度美术
	•	可使用简单几何体或低模资源搭建
	•	重点是路径、视线与掩体布局合理

⸻

6.5 敌人 AI

6.5.1 数量
	•	MVP 建议 1~3 个敌人

6.5.2 基础行为
敌人至少包含以下状态：
	•	待机 / 巡逻
	•	发现玩家
	•	寻找射击位置
	•	瞄准 / 发射
	•	受击 / 死亡

6.5.3 感知逻辑
	•	敌人在一定距离内可发现玩家
	•	视线被掩体阻挡时，发现能力下降或失效
	•	玩家长期暴露更容易被锁定

6.5.4 攻击逻辑
	•	敌人使用与玩家相同或简化版的弓箭攻击
	•	敌人攻击不要求过于复杂，但要体现蓄力后发射
	•	可加入简单射击间隔，避免高频不公平攻击

6.5.5 移动逻辑
	•	敌人可在掩体之间进行有限移动
	•	MVP 不要求高级寻路系统，但至少要避免明显卡死

⸻

6.6 UI 与反馈

6.6.1 HUD
至少包含：
	•	玩家血量
	•	敌人剩余数量或状态
	•	准星
	•	当前蓄力反馈

6.6.2 命中反馈
	•	命中敌人时应有明显反馈
	•	可包含：受击闪烁、血量变化、简单音效、命中特效

6.6.3 状态反馈
	•	蹲下状态有视觉提示
	•	探头状态有镜头或 UI 提示
	•	胜负状态有明显提示

⸻

7. 非功能需求

7.1 性能
	•	在主流桌面浏览器中可稳定运行
	•	初始目标帧率：30 FPS 以上，理想为 60 FPS
	•	地图规模与模型数量应控制在 MVP 需求内

7.2 可维护性
	•	模块化结构，便于后续增加联机或更多敌人
	•	角色控制、战斗、AI、UI、地图分层实现

7.3 可扩展性

后续应便于扩展：
	•	更多地图
	•	更多敌人类型
	•	更复杂动画
	•	联机模式
	•	缩圈或生存机制

⸻

8. 交互设计草案

8.1 键位建议
	•	W / A / S / D：移动
	•	鼠标：视角控制
	•	右键：进入瞄准
	•	左键按住：蓄力
	•	左键松开：发射
	•	Ctrl 或 C：蹲下
	•	Q：左探头
	•	E：右探头
	•	R：重开（可选）
	•	Esc：暂停（可选）

8.2 输入优先级建议
	•	瞄准状态下优先处理射箭逻辑
	•	探头优先依赖掩体附近场景
	•	非瞄准状态下不允许误触发发射

⸻

9. 范围分层

9.1 必须做（MVP 核心）
	•	第三人称移动
	•	基础镜头跟随
	•	小型静态地图
	•	简单掩体碰撞
	•	弓箭瞄准、蓄力、发射
	•	箭矢飞行与命中
	•	玩家 / 敌人血量
	•	1~3 个基础敌人
	•	胜负判定
	•	重开功能

9.2 最好做（MVP 增强）
	•	蹲下
	•	Q / E 探头
	•	基础命中特效
	•	简单敌人巡逻与掩体选择
	•	镜头防穿墙优化
	•	敌人基础受击反馈

9.3 先别做（非 MVP）
	•	多人联机
	•	缩圈
	•	搜物资
	•	背包
	•	武器切换
	•	复杂地形
	•	大量敌人
	•	高级动画融合
	•	排行榜 / 匹配

⸻

10. 关键风险

10.1 镜头体验风险

第三人称掩体玩法中，摄像机极易出现遮挡、穿模、观察不清的问题。

10.2 探头判定风险

若摄像机位置、角色位置、箭的发射点不一致，会造成“看起来能射，实际撞墙”的违和感。

10.3 掩体战成立性风险

若地图掩体布局不合理，玩家可能只会站桩互射，掩体系统无法体现价值。

10.4 AI 复杂度风险

敌人若过笨，游戏缺乏压力；若过强，玩家缺乏操作空间。MVP 应先追求可验证，不追求高智能。

⸻

11. 开发阶段建议

阶段 1：基础移动与镜头

目标：验证第三人称控制是否顺畅。

交付：
	•	角色移动
	•	摄像机跟随
	•	简单地图漫游

阶段 2：弓箭射击

目标：验证射箭手感与命中判定。

交付：
	•	瞄准
	•	蓄力
	•	发射
	•	箭飞行与命中

阶段 3：掩体玩法

目标：验证墙、石头、树木能否支持博弈。

交付：
	•	掩体碰撞
	•	蹲下
	•	Q / E 探头
	•	命中遮挡

阶段 4：敌人 AI 与胜负

目标：形成完整的单局对抗流程。

交付：
	•	基础敌人逻辑
	•	血量
	•	胜负界面
	•	重开

⸻

12. 验收标准

MVP 验收通过需满足：
	1.	玩家能在桌面浏览器中进入场景并流畅控制角色。
	2.	地图中存在多个可作为掩体的静态物体。
	3.	玩家可进行瞄准、蓄力和发射弓箭。
	4.	箭矢具有飞行过程，并可被掩体阻挡。
	5.	玩家可使用蹲下与至少一种探头机制进行战斗。
	6.	敌人可被发现、攻击并死亡。
	7.	敌人能够对玩家造成伤害。
	8.	游戏可明确结束，并支持重新开始。

⸻

13. 后续扩展方向

在 MVP 验证通过后，可考虑扩展：
	•	更细腻的角色动画
	•	更完整的掩体切换系统
	•	多种敌人行为
	•	更丰富地形与关卡
	•	联机 1v1 / 2v2 模式
	•	生存模式或缩圈模式

⸻

14. 一句话定义

这是一个基于 Three.js 的网页端第三人称小地图战术射箭对抗游戏 MVP，核心验证目标是：掩体 + 蹲下 / 探头 + 弓箭抛物线射击 是否能形成有趣、可扩展的战斗体验。
